Key findings

01 — Horizontal partitioning
Partition key choice is everything
Partitioning by id range provided minimal benefit. Partitioning by year_reported enabled partition pruning and dropped a 78-row query from 0.183s to 0.001s — a 180× speedup.
02 — Vertical partitioning
NULL rate drives the benefit
weap_cd has a 50%+ NULL rate. After vertical partitioning, targeted queries on that column improved by over 300%. Columns with high NULLs and high correlation are the best candidates.
03 — Indexing
Single biggest win — 300%+ gain
Discovered accidentally: benchmarks were slow until the team realized indexes hadn't been applied. Adding indexes on weap_cd, crim_cd, and premis_cd resolved performance across all configurations.
04 — ENGINE=MEMORY
Disappointing — on par or slower
Memory-mapped tables consumed 997MB and 637MB respectively, hitting MySQL's 1GB table limit. Performance was on par with or slower than InnoDB in most cases. Not recommended at this dataset size.
05 — Compression
50% smaller, zero perf change
ROW_FORMAT=COMPRESSED reduced the original table from 104MB to 49MB and the vertical table from 69MB to 35MB. No measurable query performance change — purely a storage win driven by VARCHAR column variety.
06 — Convergence effect
More rows = smaller differences
As result row count increases, query time differences between configurations converge. The benefit curve resembles a bounded square root — most dramatic at low row counts, diminishing at full-table scans.
⚠
ENGINE=CONNECT overhead

Federated table queries via ENGINE=CONNECT added 8+ seconds of overhead to SELECT * due to network round-trips. Viable only for data that is almost never retrieved.

Foreign key satellite tables

-- Core table (victim + incident info)

  FOREIGN KEY (id) REFERENCES vpart_crime(id));
-- Indexes on satellite tables
CREATE INDEX crim_cd_index ON vpart_crimecd_table (crim_cd);
CREATE INDEX weap_cd_index ON vpart_weap_table (weap_cd);
CREATE INDEX premis_cd_index ON vpart_premis_table (premis_cd);

Data loading — inserts & PHP import script

ℹ
Import approach

Data was loaded from the LA Crime CSV via a PHP PDO script using prepared statements and transaction batching. The same pattern was repeated for each partitioned table variant.

PHP import script

PDO batch insert with transaction

try {
    $pdo = new PDO($dsn, $user, $pass);
    $pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

    $file = fopen("LA_Crime_Data_2023_to_Present_data.csv", "r");
    fgetcsv($file); // skip header row

    $start = microtime(true);

    $stmt = $pdo->prepare("
        INSERT INTO crime (
            id, dr_no, date_reported, date_occ, time_occ,
            area, area_name, rpt_dist, part_1_2, crim_cd, crim_desc,
            mocodes, vict_age, vict_sex, vict_desc,
            premis_cd, premis_desc, weap_cd, weap_desc, status, status_desc
        ) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
    ");

    $pdo->beginTransaction();
    while (($data = fgetcsv($file)) !== FALSE) {
        // Convert empty strings to null before inserting
        $data = array_map(fn($v) => $v === "" ? null : $v, $data);
        $stmt->execute($data);
    }
    $pdo->commit();
    fclose($file);

    $elapsed = number_format(microtime(true) - $start, 2);
    echo "Import complete in {$elapsed}s\n";

} catch (PDOException $e) {
    echo "Error: " . $e->getMessage();
}

Populate fact table + lookup tables using DISTINCT

  area, rpt_dist, part_1_2, crim_cd, mocodes, vict_age, vict_sex, vict_desc,

  vict_age, vict_sex, vict_desc, premis_cd, weap_cd, status

INSERT INTO new_vweap_table   (weap_cd, weap_desc)   SELECT DISTINCT weap_cd, weap_desc FROM crime;
INSERT INTO new_vcrimecd_table (crim_cd, crim_desc)  SELECT DISTINCT crim_cd, crim_desc FROM crime;
INSERT INTO new_vpremis_table  (premis_cd, premis_desc)
  SELECT DISTINCT premis_cd, premis_desc FROM crime WHERE premis_cd IS NOT NULL;
INSERT INTO new_varea_table   (area, area_name)
  SELECT DISTINCT area, area_name FROM crime WHERE area IS NOT NULL ORDER BY CAST(area AS UNSIGNED);
INSERT INTO new_vstatus_table (status, status_desc)
  SELECT DISTINCT status, status_desc FROM crime WHERE status IS NOT NULL;

Compression — InnoDB ROW_FORMAT=COMPRESSED

~50% size reduction with no performance change

-- Original table: 104.16 MB → 49.60 MB
ALTER TABLE crime ENGINE=INNODB ROW_FORMAT=COMPRESSED;

-- Vertical type 2 fact table: 69.13 MB → 35.07 MB
ALTER TABLE new_vpart_crime ENGINE=INNODB ROW_FORMAT=COMPRESSED;

-- Lookup table (small to begin with)
ALTER TABLE new_vpremis_table ENGINE=INNODB ROW_FORMAT=COMPRESSED;
-- 0.05 MB → 0.04 MB
Why it works: the crime table has many VARCHAR columns of highly varying lengths.
InnoDB page-level compression exploits that variable-length data very effectively.

Written report

Final state

This project benchmarked seven MySQL table configurations against 390,088 rows of real LA crime data (2023–2025). The configurations tested were: the original InnoDB table, horizontal partitioning by ID range, horizontal partitioning by year_reported, vertical partitioning using foreign key satellite tables (type 1), vertical partitioning using normalized lookup tables (type 2), a memory-mapped version of the original table, and a memory-mapped version of the vertical type 2 fact table.

The report covers INSERT strategies, CREATE table definitions, SELECT benchmarks, table size analysis, indexing, InnoDB compression, ENGINE=CONNECT federation, and ENGINE=MEMORY behavior.

Progress from week 14

All partitioned tables and their query benchmarks were already complete heading into week 15. The final phase explored memory mapping and InnoDB compression as additional optimization avenues. Memory mapping produced inconsistent and largely disappointing results. Compression halved storage with no measurable query performance change — a useful finding in its own right.

Comparison to proposal

All proposed techniques were completed: vertical partitioning, horizontal partitioning, memory mapping, and ENGINE=CONNECT. InnoDB compression was an unplanned addition that proved valuable. The one area that fell short of original ambitions was broader CRUD testing with mock data — time constraints limited the benchmark set to SELECT queries only.

Most valuable insights — general

  • No universal best strategy. Every technique involves tradeoffs. Studying the actual data — its NULL rates, cardinality, and access patterns — is the prerequisite to effective optimization.
  • Convergence effect. As result row count increases, query time differences between configurations converge. The benefit curve resembles a bounded square root — most dramatic at low row counts, diminishing at full-table scans. This effect is especially pronounced with horizontal partitioning.
  • Partition key selectivity matters enormously. Partitioning by a non-selective key (id range) provided minimal benefit. Partitioning by a key that matches query predicates (year) enabled partition pruning and produced the project's best result: 0.183s → 0.001s on a 78-row query.

Most valuable insights — specific

  • Vertical partitioning. Columns with high NULL rates and high inter-column correlation are the strongest candidates. weap_cd has a 50%+ NULL rate and showed 300%+ improvement on targeted queries after partitioning. The key distinction between type 1 and type 2: type 2 uses small lookup tables with only distinct values, which are far cheaper to join than type 1's full-row-count satellite tables.
  • ENGINE=MEMORY. A significant disappointment. Performance was on par with or slower than InnoDB for most queries, while consuming 997MB and 637MB respectively — hitting MySQL's 1GB table size limit. The expected dramatic speedup did not materialize. Not recommended at this dataset scale.
  • ROW_FORMAT=COMPRESSED. Halved storage on both the original and vertical fact tables with zero query performance impact. Effective because the table contains many VARCHAR columns of highly varying lengths, which compresses very efficiently at the InnoDB page level.
  • ENGINE=CONNECT. Added 8+ seconds of overhead to SELECT * due to network round-trips. Potentially viable for archival data that is almost never queried, but impractical for anything operational.
  • Indexing. The single most impactful optimization — 300%+ speedup on targeted queries. Discovered somewhat accidentally: the team ran initial benchmarks without indexes applied and couldn't explain the slow results. Once indexes on weap_cd, crim_cd, and premis_cd were added, performance aligned with expectations.
